Agent Client Protocol

ACP 是 AI 编码 Agent 与代码编辑器之间的标准通信协议，如同编程语言领域的 LSP，打破 Agent-Editor 生态碎片化。

动机与定位：为什么需要 ACP？

🔗

现有痛点

每个 Editor 都需要为每个 Agent 写定制集成，每个 Agent 也要适配每个 Editor——M×N 的组合爆炸。

🌐

ACP 的目标

定义标准接口，Agent 与 Editor 只需各实现一次，即可与整个生态互操作，类比 LSP 解决了语言服务器的问题。

🎯

核心假设

用户主要在编辑器中工作，Agent 是辅助工具。协议以编辑器为中心，Agent 以子进程或远程服务形式接入。

🔓

开放生态

双方可在协议框架内独立创新，用户自由选择最佳工具组合，避免厂商锁定。

Prompt Turn 详解

一次 Prompt Turn = 从用户发送消息到 Agent 完成响应的完整交互周期

传输层

🏠

本地（stdio）

Editor 以子进程方式启动 Agent，通过 stdin/stdout 进行 JSON-RPC 通信。延迟低、无网络依赖，是默认推荐模式。一个连接可承载多个并发 Session。


已稳定 低延迟 无需认证
☁️

远程（HTTP / WebSocket）

Agent 托管在云端或分布式环境，通过 HTTP 或 WebSocket 与 Editor 通信。适用于团队共享 Agent、无本地计算资源场景。


开发中 WIP 需认证 适合远程托管

由于 Agent 通常只通过 stdio 支持 MCP，Editor 提供一个小型代理服务器将 MCP 请求通过 ACP 隧道转回自身，实现 ACP ↔ MCP 的协议桥接。

ACP vs MCP vs LSP 对比

维度 ACP（Agent Client Protocol） MCP（Model Context Protocol） LSP（Language Server Protocol）
目标 Editor ↔ AI Agent 通信标准 LLM ↔ 工具/上下文提供者 Editor ↔ 语言服务器
协议基础 JSON-RPC 2.0 JSON-RPC 2.0 JSON-RPC 2.0
交互模式 双向、流式、会话驱动 请求-响应为主 请求-响应 + 通知
传输层 stdio（稳定）/ HTTP+WS（WIP） stdio / SSE / HTTP stdio / Socket
特色内容 Diff 可视化、Agent 计划、权限申请 工具描述、资源、提示模板 自动补全、诊断、重构
关系 复用 MCP 类型，可桥接 MCP ACP 可透传 MCP 配置给 Agent 独立协议，ACP 借鉴其理念
